# Content Type Selector — Design Spec

## Summary

Add a content type dropdown to the session creation form that customises what the web search context feature looks for. Ships with three presets (Football Match, Football Podcast, General). Users can add custom types — the server generates a prompt hint from the label via a single Haiku call.

## Data Model

A `content-types.json` file in the project root stores all types (presets + custom):

```json
[
  {
    "id": "football-match",
    "label": "Football Match",
    "promptHint": "full matchday squads (starting XI + bench), managers, competition, venue"
  },
  {
    "id": "football-podcast",
    "label": "Football Podcast",
    "promptHint": "host names, guest names, topics discussed, teams/players likely mentioned"
  },
  {
    "id": "general",
    "label": "General",
    "promptHint": "key people, topics, terminology, and proper nouns relevant to this content"
  }
]
```

- IDs are auto-generated from the label (slugified)
- Custom types are appended to the same file
- Prompt hints for custom types are generated by a single Haiku call (no web search): "Given a content type called '{label}', what context should I search for to help with live Italian transcription?"

## API

### `GET /api/content-types`

Returns the full array from `content-types.json`.

### `POST /api/content-types`

Request: `{ "label": "Opera" }`

Server:
1. Slugifies label to create ID
2. Returns 409 if a type with the same ID already exists
3. Makes a Haiku call to generate prompt hint from the label
4. Appends new type to `content-types.json`
5. Returns the new type object `{ id, label, promptHint }`

### `POST /api/context-search` (modified)

Request: `{ "query": "PSG vs Liverpool", "promptHint": "full matchday squads..." }`

The `promptHint` field is optional. When present, the prompt template uses it to customise what context the web search looks for. When absent, falls back to the existing generic prompt.

Updated prompt template:
```
I'm building context for a live Italian transcription tool.
Search for context about: "{query}"

Find: {promptHint}

Return ONLY valid JSON. Adapt the structure to the content type. For example, a football match might use:
{
  "match": "...",
  "competition": "...",
  "venue": "...",
  "managers": ["..."],
  "teams": [{ "name": "...", "squad": ["..."] }]
}

But a cooking show might use:
{
  "show": "...",
  "host": "...",
  "dishes": ["..."],
  "ingredients": ["..."]
}

Use whatever fields best capture the context for this content type.
Use names as they would appear on official sources.
Do not include generic vocabulary or anything not sourced from the web search.
```

Note: because the JSON structure varies by content type, `buildContextString` needs a generic fallback that iterates over all top-level keys — not just the football-specific fields. Arrays get joined with commas, strings render as-is, objects with `name`/`squad` fields use the existing team format.

## Frontend

### Session creation form layout

```
[Session name: PSG vs Liverpool, Champions League    ]
[Content type: Football Match ▼]  [🔍 Search for context]
[Context textarea                                     ]
[Create] [Cancel]
```

- Native `<select>` dropdown, populated from `GET /api/content-types` on page load
- Last option is "+ Add new..."
- Default selection: first item (Football Match)

### Adding a new content type

- Selecting "+ Add new..." shows an inline text input below the dropdown
- User types the label, hits Enter or clicks "Add"
- "Adding..." loading state on the label
- `POST /api/content-types` with the label
- On success: new type appears in dropdown and is auto-selected, input hides
- On failure: error shown, input stays visible

### Search flow

When toggle is flipped on:
1. Read selected content type's `promptHint`
2. Read session name as query
3. `POST /api/context-search` with `{ query, promptHint }`
4. Populate context textarea with result

Same behaviour as the existing toggle — just adds the prompt hint.
